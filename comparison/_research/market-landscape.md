# Market Landscape

This note is narrative synthesis for the comparison program. It is useful context, but it is not canonical product data.

## Category Narratives

### AI coding IDEs

The main editor-first tools are still editor products with agents layered on top. Their strength is editing depth and code-assist ergonomics, not board-oriented task orchestration.

The persistent gap versus SlayZone is task management. Users still need a separate system of record or planning layer even when the editor itself becomes more agentic.

### AI CLI agents

CLI agents are strong execution engines but weak coordination surfaces. They can run serious work, but they do not provide a visual task layer, per-task browser context, or a board for parallel execution management.

This means they are best understood as core engines SlayZone can orchestrate, not as products that solve the same coordination problem by themselves.

### Agent orchestrators

This remains the closest product category to SlayZone. The most relevant competitors are the products that combine agents with a multi-workspace or multi-task environment rather than just shipping an editor or a headless CLI.

Within this category, the real competitive question is no longer "can the product run multiple agents?" That is becoming expected. The harder differentiation now sits in task UX, review workflow, local-first posture, browser/editor integration, and how well the product keeps parallel work legible.

### Traditional PM tools

Traditional PM products overlap on planning, status, and collaboration, but they still do not own the execution environment. They are evolving toward agent integrations and MCP support, but not toward per-task dev environments with real terminals and isolated workspaces.

That makes them strategically important, but mostly as adjacent substitutes rather than direct product-shape competitors.

### Dev productivity and terminal-adjacent tools

Terminal and productivity tools matter because they can capture pieces of the workflow SlayZone wants to own, especially around local execution, browser context, and agent ergonomics.

Even when they are not direct competitors, they can influence user expectations for what the execution layer should feel like.

## Competitive Narratives

- **VibeKanban** is the clearest board-first threat. It validates the market for kanban plus agents, but makes different tradeoffs around web delivery, infra complexity, and native desktop behavior.
- **Superset.sh** is the closest architectural neighbor. It is strong on terminal, browser, and editor surfaces, but weak on task management and board orchestration.
- **Conductor** is a serious orchestrator threat if it deepens task management. Its current position is more workspace-and-agent centric than board centric.
- **Maestro** is a power-user launcher rather than a board product. It matters because it shows how much value advanced users place on real PTY orchestration.
- **Cursor, Windsurf, and Zed** compete for the default coding seat, but they do not solve the same task-orchestration problem.
- **Linear and Jira** remain strong systems of record. Their threat is not local execution, but continued convergence toward AI-aware planning and assignment.
- **Claude Code and Codex** are first-party execution anchors. They are not merely inputs to the ecosystem; they are also substitutes for teams that do not feel enough pain to adopt a higher-level cockpit.

## Trends

1. Agent orchestrators are the fastest-moving comparison category.
2. Multi-agent support is becoming standard; UX quality is the real battleground.
3. Git worktree isolation is increasingly expected rather than differentiating.
4. PM tools are adding AI agents and MCP features, but not terminals.
5. Value is shifting upward from raw agents to orchestration, review, and workflow design.
